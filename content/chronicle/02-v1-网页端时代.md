## 技术架构

```
用户浏览器
    │
    ▼
Cloudflare Tunnel（HTTPS 终结）
    │
    ▼
nginx（反向代理 + 静态文件）
    │
    ▼
gunicorn（Django WSGI）
    │
    ├── accounts（用户/认证）
    ├── courses（讲义/课程）
    ├── interactions（答题/评分）
    ├── qbank（题库管理）
    ├── system（系统配置）
    └── math_platform（项目配置/URL 分发）
```

**关键特征：**
- **纯服务端渲染：** Django Template + HTML/CSS/JS，无 SPA
- **在线依赖：** 每次操作都需要网络请求，无离线能力
- **单数据库：** SQLite 单文件，所有数据集中存储
- **Markdown + LaTeX：** 问答区使用 `marked.js` + `KaTeX` 前端渲染

### 教师端（v1）

v1 没有独立的教师端 App。教师通过 Django Admin 后台 + 自定义管理页面完成：
- 讲义上传与编辑
- 学生进度查看
- 班级管理

## 数据基础

### 题库

- **总量：** ~798 道北京高考数学题
- **年份覆盖：** 2020 ~ 2026
- **来源：** 一模、二模、高考真题

## v1 的主要问题（催生 v2 的原因）

| 问题 | 表现 | v2 的改进 |
|------|------|----------|
| **移动端体验差** | 网页在手机上缩放/点击困难 | Flutter 原生跨平台 |
| **网络依赖** | 无网就不能用 | 离线优先（SQLite + 同步引擎） |
| **慢速渲染** | 题库页面加载大量 DOM | Flutter 原生渲染 |
| **单端** | 只有学生浏览器可用 | 学生端 + 教师端双 Flutter App |
| **无系统化测试** | 没有 CI，全靠手动 | GitHub Actions + 分层测试体系 |
| **无审计流程** | bug 改完就当修好了 | 5 维度审计 + fix-batch-workflow |

---

## v1 的 GitHub Actions CI

v1 后期（6/24 左右）引入了 GitHub Actions CI，但只覆盖了 Django：
- `flake8` Python 代码风格检查
- `pytest` 单元测试（少量）

v2 的 CI 在此基础上扩展为并行 Flutter analyze + Django flake8/pytest。
